# Mission Control 面向非技术用户路线图

## 目标

把 Mission Control 从“给开发者操作 agent 的看板”升级成“不会编程的人也能指挥 OpenClaw 完成任务的工作台”。

核心标准不是功能多少，而是：

- 用户能不能只用自然语言下达任务
- 用户能不能看懂系统现在在做什么
- 用户能不能在卡住时知道自己下一步该做什么
- 用户能不能在不懂代码的情况下完成设计、执行、验证闭环

## 当前现状

Mission Control 已经具备以下基础能力：

- 看板式任务管理
- OpenClaw coder 自动派单
- QA / browser verification 自动派单
- 任务详情里的 transcript、截图、证据
- clarification / plan / verification 的雏形
- docs memory 写入能力

但它仍然更像一个工程控制面板，而不是非技术用户产品。

## 当前距离“我妈可用”还差什么

### 1. 任务创建仍然偏工程化

当前问题：

- 用户仍然需要理解任务卡字段
- `targetUrl`、acceptance criteria、lane 等概念偏技术
- 用户需要自己组织任务描述

目标：

- 改成聊天式任务创建器
- 系统自动从自然语言补全 title / objective / acceptance criteria
- 系统自动判断是否需要 browser QA

### 2. 状态文案不够人话

当前问题：

- `Backlog / Triaged / In Progress / Testing` 更像工程流程
- 非技术用户不容易理解当前阶段

目标：

- 把状态改成用户语言
- 让用户一眼知道“正在做”“等我回答”“等我确认”“已完成”

### 3. 结果表达还不够产品化

当前问题：

- 系统已经有 verification harness，但结果仍偏技术
- 用户难以快速理解 fail / partial / evidence / repro steps

目标：

- 结果卡片只表达三件事：
  - 结果是什么
  - 为什么这样
  - 用户下一步该点什么

### 4. Browser QA 还不够“可见、可接管”

当前问题：

- 虽然已支持 browser QA 和截图
- 但用户还不能强烈感知“agent 正在浏览器里替我做事”

目标：

- 显示访问了哪个页面、抓到了哪些截图、卡在哪
- 支持重试、纠正页面、人工接管

### 5. 错误恢复缺少引导层

当前问题：

- 出错时仍偏工程日志与任务流转
- 非技术用户不容易知道自己需要做什么

目标：

- 把错误翻译成下一步动作
- 例如“我需要你登录”“我找不到正确页面”“请回答这个问题后我继续”

## 产品方向判断

现阶段优先做：

- OpenClaw-first 的 Mission Control

不优先做：

- 通用多-agent 平台

原因：

- 当前目标是“让不会编程的人 leverage OpenClaw”
- 当前架构已经深度围绕 OpenClaw 建立
- 现阶段更缺产品完成度，而不是 provider 抽象层

原则：

- 先产品，后平台
- 先把 OpenClaw 版本做到顺手，再抽象通用能力

## 版本路线图

## V1：能独立使用

目标：

- 用户可以自己创建任务、看懂状态、知道下一步做什么

### [x] V1.1 聊天式任务创建器

用户只需要回答：

- 你想做什么
- 这是改外观、修问题，还是做新功能
- 要不要我自动去浏览器检查
- 有没有具体页面

系统自动生成：

- title
- objective
- acceptance criteria
- browser QA 配置
- 默认 verification harness

### [x] V1.2 人话状态

建议把内部工程状态映射成用户可见状态：

- 想法
- 准备执行
- 正在处理
- 等你确认
- 已完成
- 需要修复

### V1.3 结果卡片

每张任务结束后显示：

- 结果：完成 / 需要补充 / 失败
- 原因：一句话
- 下一步按钮：继续修、回答问题、重新检查

### V1.4 clarification 变成按钮问答

要求：

- 少问
- 短问
- 给推荐项
- 默认尽量自动填

非目标：

- 让用户写长段技术说明

## V2：像数字助理

目标：

- 用户开始把它当作“会主动做事的助理”

### V2.1 Browser QA 可视化

展示：

- 打开了哪个页面
- 抓了哪些截图
- 最后停在哪一步

操作：

- 再检查一遍
- 不是这个页面
- 我来接管

### V2.2 引导式失败恢复

把工程错误翻译成用户动作：

- 我需要你登录
- 我找不到正确页面
- 我已经改好了，但浏览器检查没通过

### V2.3 一键模式

提供高层任务模式，而不是暴露内部 agent 角色：

- 帮我设计
- 帮我修
- 帮我检查
- 帮我整理文档

### V2.4 常用任务模板

例如：

- 改网站文案
- 调整页面设计
- 修网页 bug
- 写一份说明文档

## V3：长期愿意用

目标：

- 用户形成习惯，长期把它当工作台

### V3.1 偏好与长期记忆

记住：

- 常用页面
- 常见任务类型
- 偏好的表达方式
- 常见验收习惯

### V3.2 每日工作台

首页提供：

- 今天该做什么
- 哪些任务卡住了
- 哪些需要用户点一下

### V3.3 放心感设计

所有流程都要明确回答：

- 我现在在做什么
- 为什么停下了
- 你只需要做什么

## 当前优先级最高的 3 件事

### [x] Priority 1：聊天式建任务

原因：

- 这是所有非技术用户体验的入口
- 如果入口还像工程单据，后续流程再好也会挡人

### [x] Priority 2：人话状态与结果页

原因：

- 用户需要理解流程
- 用户必须知道结果和下一步

### Priority 3：Browser QA 可视化 + 一键重试

原因：

- 这是“它真的在替我做事”的最强感知来源
- 也是网页类任务里最核心的信任建立方式

## 设计原则

### 原则 1：默认替用户做决定

只在真正关键时提问。

### 原则 2：所有技术状态都要翻译成人话

内部可以保留工程状态，用户界面只展示用户语言。

### 原则 3：失败必须带下一步

不能只说失败，要告诉用户该点什么。

### 原则 4：让用户觉得自己在“指挥助理”

而不是在“操作开发工具”。

## 成功标准

当满足以下条件时，可以认为接近“我妈可用”：

- 用户可以不理解代码，也能创建任务
- 用户可以不理解 agent 内部流程，也知道当前进度
- 用户可以在系统卡住时完成最小必要交互
- 用户可以通过截图、结果卡片、下一步按钮继续推进任务
- 用户不需要学习 prompt engineering

## 下一步实施建议

先做一个最小闭环：

1. 聊天式任务创建器
2. 人话 lane / 状态映射
3. 用户结果卡片
4. browser QA 可视化

完成后再继续做：

5. 引导式失败恢复
6. 一键模式
7. 模板与长期记忆
